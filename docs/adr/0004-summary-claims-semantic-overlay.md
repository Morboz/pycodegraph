# Summary Claims as an external semantic overlay with independent storage

We introduce a semantic overlay for LLM-derived propositions about code — Summary Claims — stored independently from the deterministic hard-structure graph. Claims are generated by an external process (an LLM summarizer that reads pycodegraph's hard-structure query API) and written back through a storage/retrieval API; pycodegraph itself remains deterministic and LLM-free. We chose fully independent storage (a `summary_claims` table plus a `claim_grounding` table with line-range endpoints) over adding new NodeKind/EdgeKind entries to the core `nodes`/`edges` tables because a Summary Claim is not a Symbol (it has no `qualified_name`, `file_path`, `language`, or source position), the `edges` table endpoints are `ForeignKey("nodes.id")` so a claim cannot be an edge source, and embedding probabilistic LLM-derived facts in the deterministic graph would break the reproducibility guarantee that CONTEXT.md and ADRs 0001–0003 rely on. MVP scope is a load-once smoke test: no claim regeneration, idempotency, or promotion; claims are retrieved by natural-language full-text search (`search_claims_fts`, mirroring `search_nodes_fts`) and returned bundled with their grounding spans. Python-focused motivating material (Ansible module surfaces); language-agnostic storage.

## Considered Options

- **New NodeKind/EdgeKind entries in the core `nodes`/`edges` tables**: Rejected — a Summary Claim is not a Symbol (no `qualified_name`, `file_path`, `language`, line position); the `edges` table endpoints are `ForeignKey("nodes.id")` so a claim cannot be an edge source; and embedding probabilistic LLM-derived facts in the deterministic graph breaks the reproducibility guarantee the rest of the system is built on (the "summary reverse-pollutes the KG" failure mode).
- **Seven distinct node kinds (SummaryClaim, FeatureConcept, ArchitectureComponent, BehaviorContract, InterfaceSurface, TaskShapeHint, EntityAlias)**: Rejected — without a consumer that distinguishes them, their boundaries are speculative over-abstraction. Collapsed to a single `SummaryClaim` row with a `claim_type` discriminator field. `EntityAlias` is a deterministic symbol-normalization concern belonging to the resolution layer, not the semantic overlay, and is tracked separately.
- **Nine distinct edge kinds**: Rejected — after the node-type collapse every relation is "a claim points at a code location with a relation flavor," so they unify into one `claim_grounding` table with a `relation` discriminator (the same collapse shape applied to edges). Provenance/lifecycle relations (`derived_from`, `stale_after_commit`, `contradicted_by`) are columns on the claim row, not edges.
- **Vector / embedding retrieval**: Rejected for now — a vector index would import a non-deterministic embedding-model dependency into pycodegraph, contradicting the "claims are generated externally; pycodegraph stays deterministic" boundary. Full-text search already exists per-backend (`search_nodes_fts`) and is cloned for claims with a stemming tokenizer (`porter` on SQLite, `english` on PostgreSQL), since claim text is natural language and unstemmed FTS would silently miss paraphrased queries (e.g. "decompression" vs "decompress"). Embeddings, if ever needed, are the external generator's responsibility.
- **Claim generation inside pycodegraph (an LLM in the indexing pipeline)**: Rejected — would destroy the deterministic, reproducible property the rest of the system is built on. Generation lives in an external process that reads pycodegraph's hard-structure query API and writes claims back through the storage API.
- **Proposition-level upsert and a promotion state machine (soft → hard)**: Rejected for the MVP — an LLM rephrases the same proposition across runs, so a claim has no stable cross-regeneration identity; proposition-level upsert is a false promise, and promotion requires exactly that stable identity. Deferred until a real validation consumer exists. The MVP is load-once (`clear_claims` + bulk insert).
